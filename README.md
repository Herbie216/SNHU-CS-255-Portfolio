# SNHU-CS-255-Portfolio
CS 255 Portfolio Reflection – DriverPass Project

Project Summary:

The DriverPass project was based on a client who wanted to improve driver training and help students pass their DMV tests. The client, DriverPass, needed a system that could handle online learning, practice tests, and scheduling in-person driving lessons. The system also needed to support multiple users, including customers, staff, and administrators, while keeping everything organized and secure. Overall, the goal was to design a system that was easy to use, accessible online, and flexible enough to grow in the future.

What I Did Well:

One thing I feel I did particularly well was organizing the system using UML diagrams. Breaking the system down into use case, activity, sequence, and class diagrams helped me better understand how everything connects. I also focused on keeping the designs clear and readable instead of overcomplicating them. That made it easier to follow the flow of the system and ensured that the design matched the client’s needs.

What I Would Revise:

If I could revise one part of my work, I would spend more time refining the diagrams earlier in the process. As I continued working, I found small improvements I could have made to the structure and flow. Going forward, I would review and revise diagrams more often instead of waiting until later stages. This would help improve accuracy and save time in the long run.

Interpreting User Needs:

I interpreted the user’s needs by carefully reviewing the interview transcript and identifying the key features the system needed to support, such as scheduling lessons, managing accounts, and tracking activity. I then translated those needs into system components and workflows using UML diagrams. Considering the user’s needs is important because the system should be built around how people actually use it, not just how it works technically. If the design does not match the user’s expectations, the system will be difficult to use and less effective.

My Approach to System Design:

When designing software, I like to break things down step by step and build from a high-level view into more detailed components. Starting with use cases helps define what the system needs to do, and then activity and sequence diagrams help map out how it works. The class diagram helps define the structure behind everything. In the future, I would continue using this approach and also incorporate strategies like test-driven development, since I am very fond of that method. It helps catch issues early and keeps the system more reliable as it grows.
